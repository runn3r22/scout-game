Scout Game.

Gamified alpha sourcing for the Fair autonomous fund

Scouts tag @fairvc on X with token alpha. Fair's AI agent evaluates, scores, and acts — saving to DB, publishing signals, or executing trades. Scouts earn $FAIR rewards through a proportional points system. Better signals, bigger rewards. Hold tokens, climb tiers, earn more.

The scout game turns Fair's community into a distributed research team. Human pattern recognition + AI evaluation + on-chain execution.

1.2 Key Design Decisions

Same database. Scout submissions hit the same projects and agent_memory tables as the autonomous pipeline. Duplicate info = rejected.

Same evaluation frameworks. PLAYBOOK (Metagame Theory, Attention Theory, Probabilistic Thinking, entry checklist) applies equally to scout and pipeline signals.

Signal weight ≠ evaluation bias. GP's submission gets processed sooner, but confidence score comes from fundamentals, not scout's tier. T1 scout with genuine 80%+ project = same trade outcome as GP.

1.3 How Scouts Improve the Agent

Coverage expansion: Scouts on X extend the agent's reach — different sources, networks, pattern recognition.

Human intuition: "This dev's wallet did something weird," "this team built X before," "this narrative is about to rotate" — hard to encode algorithmically.

Database enrichment: Even rejected submissions create signal/noise profiles, helping tune pipeline scoring over time.

Feedback Mechanisms

Track record per scout (post-MVP): Each scout builds history in scout_submissions. Scout A: 40 subs, 12 accepted, 3 profitable trades → high-signal. Scout B: 60 subs, 5 accepted, 0 trades → noise.

Agent performance tracking: When agent trades on scout signal, P&L is tracked. Can't tie to individual rewards (parked — legal), but CAN: display aggregate stats ("Scout-sourced trades: X% win rate"), recognize top scouts ("Scout of the Week"), evaluate channel value vs pipeline.

1.5 The "Paid Evaluation" Feature

A holder of token X buys $FAIR → becomes scout → submits token X. This is intentional and valuable:

  • Creates direct buy pressure on $FAIR from external communities
  • Agent evaluates objectively — buying $FAIR buys ACCESS, not a positive signal
  • If project is good, Fair benefits from discovering it. If bad, rejected.
  • Coordinated push (multiple scouts, same token, short window) → confidence DECREASES

Positioning: "Want Fair's AI to evaluate your project? Hold $FAIR and submit it."

1.6 AI Agents as Scouts

Flagged for future. Allow other AI agents to participate — 24/7 monitoring, "AI hiring AI" narrative. But: trivially cheap to sybil, high spam risk, potential keyword stuffing. Recommend: separate "Agent Scout" tier (50M+ minimum), separate rate limits, identity verification. Don't build for MVP.

1b — Evaluation Pipeline

⚠️ REQUIRES DEDICATED DESIGN

The evaluation pipeline is the core of the scout game. If the agent evaluates poorly — missing good signals or acting on bad ones — everything else is wasted. This needs its own dedicated design process, not just "use PLAYBOOK."

What Needs to Be Designed

1. Scout-Specific Evaluation Criteria

The existing PLAYBOOK was designed for autonomous pipeline signals (structured Farcaster casts scored by GPT-5). Scout submissions are different — tweets are short, unstructured, slang-heavy. The evaluation must handle:

  • Extracting actionable signal from informal tweet content
  • Parsing token/project references from natural language ("this dev is cooking," "check the chart")
  • Weighting scout's own analysis vs just a link drop
  • Handling multi-token or multi-project tweets

2. Quality Assessment Logic

How does the agent distinguish:

  • Genuine alpha (found something real and early) vs publicly known information
  • Keyword stuffing (triggers high confidence without substance)
  • Bag pushing (valid via paid evaluation, but agent must be objective)
  • Post-run submissions (less valuable than pre-run)

3. Real-Time Cross-Reference

For each submission, cross-reference:

  • On-chain: holder distribution, liquidity depth, deployer history, contract verification
  • Social: who else is talking, engagement authenticity
  • Timing: before or after price run?
  • Existing DB: does Fair already track this project?

4. Anti-Gaming

Once scouts understand confidence triggers, they'll optimize. Resist:

  • Keyword stuffing (all the right terms, no substance)
  • Fake social proof (bought engagement on tagged tweet)
  • Coordinated push (many scouts, same token, short window → reduce, not boost confidence)
  • Adversarial text designed to manipulate scoring

5. Learning Loop

  • Which scout signal types lead to profitable trades?
  • Which scouts consistently produce high-quality vs noise?
  • What submission patterns predict success?
  • How to weight scout reputation in future evaluations (post-MVP)

6. Confidence Calibration

Thresholds directly impact economics (lower bar = more rewards), signal quality (higher bar = fewer but better), and scout motivation (too strict = frustration). Start conservative (high bar), tune down if acceptance too low.

Next step: Dedicated design session before GP test. Define evaluation prompt, test with example tweets, red-team with adversarial submissions, set initial thresholds. This is the single most important technical workstream.

2.4 On-Chain: Hedgey Contracts + Custom Claim UI

Approach: Claim-based distribution. We publish merkle root, scouts claim when ready. They pay gas (Base ~$0.001/claim). Our cost: $0.

Why Hedgey:

  • Already in Fair ecosystem (token locks)
  • Audited by Consensys Diligence (5+ audits)
  • Deployed on Base
  • Open-source contracts on GitHub (ClaimCampaigns, DelegatedTokenClaims)
  • API + contracts are free (public goods model)

Integration: Hedgey Contracts + Our UI

ADMIN SIDE (automated, end of day/week):
  Calculate points → Generate CSV (wallet, amount)
  → Upload to Hedgey API (creates merkle tree, stores proofs)
  → Deploy claim campaign on Hedgey contract
  → Store claim UUID in Supabase

SCOUT SIDE (on fair.xyz/scout/rewards):
  Connects wallet (already via Privy)
  → Frontend queries Hedgey API for available claims
  → Shows: "You have 5,000,000 $FAIR to claim"
  → Scout clicks "Claim" → signs tx → tokens sent
  → All on our site. Hedgey contract under the hood.

Note: Hedgey team joining Anchorage Digital (acquisition). Risk limited — we use their immutable on-chain contracts + open-source code. If API goes down, we generate merkle trees ourselves.

3 — Outcomes & KPIs

3.1 Goals (Priority Order)

Priority 1: Better Token Sourcing

The entire point. Human scouts find alpha that algorithms miss. Scouts on X extend the agent's reach — different sources, different networks, different pattern recognition.

Success: Agent receives scout-sourced signals that lead to profitable trades the pipeline would have missed. Even one trade per season that the pipeline wouldn't have found independently makes the mechanic valuable.

Priority 2: Visibility on Winning Plays

If @fairvc becomes the default tag when someone finds alpha, every runner gives us visibility. When the token pumps, people see Fair was tagged early.

Success: @fairvc becomes a reflexive tag in CT alpha circles. People tag us without being scouts because they've seen it work.

Priority 3: Brand Awareness

Every submission is a public tweet mentioning @fairvc. Every published signal demonstrates capability. Leaderboard creates competition → posts about the game → awareness.

Priority 4: New Token Buyers

Every scout must hold $FAIR. Tier progression incentivizes holding and buying more. Flywheel: more scouts → more alpha → better trades → token up → more scouts.

4.3 Retention Mechanics

Layer 1: Tier Progression. Earn → hold → tier up → earn more. T1 earns 10M → holds → T2 (1.5x). Costs nothing, benefit concrete.

Layer 2: Leaderboard. Weekly top 10, public recognition, 25% of weekly pool to #1. Posted by @fairvc = competition → content.

Layer 3: Track Record (post-MVP). Scout Score: submissions accepted, signals→trades, win rate, seasons active, streaks. Money can't buy a profitable track record — creates status independent of holdings.

Layer 4: Hold Duration Multiplier (option, S2+).

Hold DurationBonusExample: T2 (1.5x)
<30 days1.0x→ effective 1.5x
30-90 days1.1x→ effective 1.65x
90-180 days1.2x→ effective 1.8x
180+ days1.3x→ effective 1.95x

Rewards loyalty, discourages buy-sell around seasons. Flagged for S2+ — tier system sufficient for MVP.

Layer 5: Badges (post-MVP). "S1 Pioneer," "Alpha Hunter" (≥3 trade signals/season), "Consistent Scout" (daily for full season), "First Blood" (first trade-level submission). Zero cost, collectors mentality.

4.4 Edge Cases

Tier drops mid-season: Tier checked at each submission. Holdings dropped → lower multiplier on new submissions. Already-earned points not adjusted.

Banned scout: Points removed from pool calculations. Unclaimed rewards returned to pool.

Pool runs fast: Proportional model = can't overspend. But if reward per point drops too low, scouts lose motivation. Option 1 (default): accept dilution. Option 2: add tokens to pool with announcement.

Disputed evaluation: No appeals in MVP. Agent's decision final. Post-MVP: "dispute" button, manual review, cap 3/season/scout.

E — Token Economics

Core Design

Proportional points system. Pool splits by weighted points each day — no fixed token amounts per action. More scouts = dilution per point but pool never overspends. Fewer scouts = concentration.

  • Daily: Points earned today → daily pool (50%) split proportionally by weighted points
  • Weekly: Cumulative weekly points → weekly pool (50%) split by leaderboard rank
  • Unused daily pool rolls into weekly pool
  • Multiple submitters on same project ALL get rewarded (signal/trade). DB saves: only new info.
  • Holdings affect signal weight (attention priority) AND point multiplier
