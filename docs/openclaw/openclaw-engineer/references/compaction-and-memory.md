# Compaction & Memory — Deep Reference

## Compaction: Two Paths, One Catastrophic

### Maintenance Compaction (Good Path)

Triggers after a successful turn when:
`contextTokens >= contextWindow - reserveTokensFloor - softThresholdTokens`

For a 200K context window with defaults (reserveTokensFloor: 20000, softThresholdTokens: 4000):
triggers at ~176K tokens.

Before summarizing, a **memory flush** runs: a silent agentic turn with two injected prompts:
- System prompt: "Session nearing compaction. Store durable memories now."
- User prompt: "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO_REPLY if nothing to store."

The agent saves important context to disk, then older history is summarized with ~20K tokens of
recent messages kept intact.

### Overflow Recovery (Bad Path)

Triggers when the model API **rejects** the request (context overflow error). Emergency
compression fires with:
- NO memory flush
- NO saving opportunity
- Maximum context loss
- Everything not already written to disk is permanently gone

### The Critical Bug: softThresholdTokens Doesn't Scale

`softThresholdTokens` defaults to 4000 and does NOT scale with context window size.

| Model Context | Default Flush Trigger | Problem |
|---|---|---|
| 200K | ~176K tokens | OK — reasonable headroom |
| 500K | ~476K tokens | Risky — API may reject before this |
| 1M | ~996K tokens | Dangerous — API will reject well before 996K |

**Production fix for large-context models:**
```json5
{
  agents: {
    defaults: {
      compaction: {
        reserveTokensFloor: 120000,
        memoryFlush: {
          softThresholdTokens: 50000
        }
      }
    }
  }
}
```

### Compaction Has Zero Fault Tolerance

Compaction is the ONLY user-facing LLM operation with no retry and no model fallback. If the
compaction model call fails, the system doesn't try again or fall back to a cheaper model.
This is tracked as issue #14543.

**Mitigation:** Override the compaction model to something reliable and cheap:
```json5
{
  agents: {
    defaults: {
      compaction: {
        model: "anthropic/claude-sonnet-4-20250514"  // saves 2-5x vs Opus
      }
    }
  }
}
```

### Compaction Modes

- **`default`**: Standard summarization — summarize old messages, keep recent ~20K tokens.
- **`safeguard`** (now default for new installs): Chunked summarization with adaptive token
  budgeting. Uses `maxHistoryShare: 0.5` to cap history's share of the context window.
  **Bug #2418:** Dropped chunks during pruning are simply removed WITHOUT summarization.

### Memory Flush Details

- Flush runs once per compaction cycle (tracked in `sessions.json`)
- Skipped when `workspaceAccess` is `"ro"` or `"none"` (sandboxed subagents can't write)
- Skipped in heartbeat runs with `lightContext: true`
- The flush is a single silent agentic turn — if the model doesn't save everything in one
  turn, the unsaved data is still lost

### Compaction Config Reference

```json5
{
  agents: {
    defaults: {
      compaction: {
        reserveTokensFloor: 20000,     // headroom floor (default)
        mode: "default",               // or "safeguard"
        model: "provider/model-id",    // optional cheaper model for summarization
        memoryFlush: {
          enabled: true,               // default
          softThresholdTokens: 4000,   // DOES NOT SCALE — adjust for your model
          systemPrompt: "Session nearing compaction. Store durable memories now.",
          prompt: "Write any lasting notes to memory/YYYY-MM-DD.md..."
        }
      }
    }
  }
}
```

### Five-Layer Compaction Resilience (Community Pattern)

From `gavdalf/openclaw-memory` — addresses the reality that compaction WILL lose information:

1. **Observer cron** (every 15 min): extracts durable facts from transcripts
2. **Reactive file watcher**: fires during heavy conversations
3. **Built-in memoryFlush**: pre-compaction hook (described above)
4. **Session startup**: loads all saved memory
5. **Session recovery**: catches missed observations during manual resets

Minimum viable defense: autojournal habit + memory flush + nightly review cron = 3 of 5 layers.

## Memory Architecture

### File Types and Loading Behavior

| File | Loaded When | Purpose |
|---|---|---|
| `MEMORY.md` | Private sessions only (bootstrap-injected) | Curated durable facts |
| `memory/YYYY-MM-DD.md` | Via tool call (NOT bootstrap-injected) | Daily append-only logs |
| `USER.md` | Every turn (bootstrap-injected) | Static user profile — survives compaction |
| Custom memory files (`memory/*.md`) | Via `memory_search` only | Domain-specific searchable memory |

Key rules:
- If both `MEMORY.md` and `memory.md` exist, only the capitalized version loads
- `memory_get` degrades gracefully on missing files — returns empty text, not an error
- Daily notes are NOT auto-generated by the framework — the LLM creates them
- At session start, the model reads today's + yesterday's daily notes via tool calls

### Memory Routing is an LLM Decision

There is no rule-based logic routing between daily notes and MEMORY.md. The system prompt
instructs the convention:
- **MEMORY.md**: "decisions, preferences, and durable facts" — curated long-term memory
- **Daily notes**: "day-to-day notes and running context" — append-only logs

The LLM decides based on its judgment of durability. This means memory quality depends directly
on SOUL.md/AGENTS.md instructions about what to save where.

### Retrieval Pipeline

```
Vector + Keyword → Weighted Merge → Temporal Decay → Sort → MMR → Top-K
```

**Hybrid search:** `finalScore = vectorWeight × vectorScore + textWeight × textScore`
- Default weights: 0.7 vector, 0.3 BM25
- Fusion is **union-based** (not intersection) — high vector + zero keyword still appears
- Both paths fetch `maxResults × candidateMultiplier` candidates (default candidateMultiplier: 4)

**BM25 scoring:** `1 / (1 + rank)` — rank 0 = 1.0, rank 9 = 0.1

**Temporal decay** (off by default):
- `decayedScore = score × e^(-λ × ageInDays)` where λ = ln(2)/halfLifeDays (default 30 days)
- 30-day-old memory = 50% of original score
- Evergreen files (MEMORY.md, non-dated memory files) are NEVER decayed

**MMR** (off by default):
- Jaccard text similarity on tokenized content (not embedding distance)
- lambda=0.7 for diversity reranking

### Fallback Chain (Memory Never Hard-Fails)

1. sqlite-vec available → cosine similarity (default)
2. sqlite-vec unavailable → JS cosine similarity in-process (slower)
3. Embeddings unavailable → BM25-only with query expansion
4. FTS5 unavailable → vector-only

### Index Storage

Location: `~/.openclaw/memory/<agentId>.sqlite`

Schema:
- `files` table: mtime/hash for change detection
- `chunks` table: text with line numbers and SHA-256 hashes
- `embedding_cache` table: cross-file deduplication
- `chunks_fts`: FTS5 virtual table (BM25)
- `chunks_vec`: vec0 virtual table (vector search)

Chunking: ~400 tokens (~1600 chars) with 80-token overlap, line-aware splitting.
File watcher: 1.5-second debounce. Sync runs on session start, search, or configurable interval.

### Embedding Providers

Auto-selection priority: **Local → OpenAI → Gemini → Voyage → Mistral → disabled**

| Provider | Model | Dimensions |
|---|---|---|
| Local | `hf:ggml-org/embeddinggemma-300m-qat-q8_0-GGUF` | varies |
| OpenAI | `text-embedding-3-small` | 1536 |
| Gemini | `gemini-embedding-001` | 768 |

- Changing providers triggers automatic full reindex (index stores provider/model fingerprint)
- Codex OAuth does NOT cover embeddings — need explicit API key
- Ollama supported but not auto-selected

```json5
  }

```

## External Memory Backends

### QMD (by Tobi Lütke / Shopify)
- Local-first search sidecar: BM25 + vectors + reranking
- Runs via Bun + node-llama-cpp with auto-downloaded GGUF models
- Config: `memory.backend = "qmd"`
- State: `~/.openclaw/agents/<agentId>/qmd/`
- Falls back to built-in SQLite if QMD fails

### Cognee
- Knowledge graph layer: extracts entities and relationships
- Graph traversal queries instead of text similarity
- Syncs on startup, auto-recalls before each run, auto-indexes after each run
- State: `~/.openclaw/memory/cognee/`

### Mem0
- LLM-based extraction: auto-determines what's worth persisting
- Memory lives OUTSIDE context window — immune to compaction
- New facts stored, stale updated, duplicates merged

### Lossless Claw (Martian Engineering)
- Replaces default compaction entirely with DAG-based summarization
- Every message persists in SQLite
- Tree structure: leaf summaries (~8 turns) → depth-1 (~32) → depth-2 (~128)
- Three compression levels: LLM summary → bullet compression → deterministic truncation
- Tools: `lcm_grep`, `lcm_describe`, `lcm_expand` for deep recall
- Config: `freshTailCount: 32` (protects last 32 messages), `contextThreshold: 0.75`
- Raw messages retained for configurable period (default 90 days)
